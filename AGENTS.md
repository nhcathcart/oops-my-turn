# Repository Guide

This file provides guidance to Claude Code when working in this repository.

## Project Overview

This repository is a full-stack starter application: a Go backend (Huma v2, PostgreSQL/pgx, Bob ORM, sql-migrate, oapi-codegen) and a React frontend (Vite, TanStack Query, shadcn/ui, Tailwind v4).

## Commands

```bash
# Start backend + frontend together (recommended for development)
make dev

# Start the API server only (port 9000, starts db if not running)
make run

# Start the frontend dev server only (port 5173)
make frontend-dev

# Start postgres and run migrations
make db-up

# Open a psql shell
make db-shell

# Apply pending migrations
make migrate-up

# Create a new timestamped migration file
make new-migration NAME=add_users

# Print the OpenAPI spec as YAML
make print-spec

# Run all code generators (migrate-up → Bob models → Go SDK → TypeScript client)
make generate

# Run backend unit tests
make test

# Run frontend unit tests
make frontend-test

# Run integration tests (uses testcontainers, no db-up needed)
make integration

# Build the frontend for production
make frontend-build

# Run all linters
make lint

# Run frontend type checker
make frontend-typecheck
```

## Adding New Endpoints

1. Create `backend/internal/server/handlers_<resource>.go` with Input/Output structs using Huma tags (`doc`, `example`, `enum`, `required`, `minimum`, `maximum`)
2. Register routes in `registerCoreRoutes()` inside a `// start <resource> region` / `// end <resource> region` block
3. Extract the authenticated user via `ctx.Value(jwtClaimsKey{}).(*JWTClaims)` — the `authenticateJWT` middleware populates this for all routes not in `unauthenticatedPaths`
4. Run `make generate` to regenerate Go SDK + TypeScript client
5. **CRITICAL: Add integration tests in `backend/internal/server/handlers_<resource>_test.go`.** Every new endpoint MUST have integration tests covering all success modes (asserting every response field: id, url, status, timestamps non-zero, nested objects, etc.) and all failure modes (asserting the expected HTTP status code). An endpoint is not complete until its integration tests pass.

**CRITICAL: Run `make generate` after any API contract change or database migration change.** This includes adding, removing, or changing routes, handler input/output shapes, OpenAPI-visible tags, and schema changes that affect generated Bob models or API clients. The change is not complete until generated artifacts are refreshed.

## Adding Database Queries

1. Create a migration: `make new-migration NAME=<name>`
2. Apply it: `make db-up` (or `make migrate-up`)
3. Use Bob ORM (`github.com/stephenafamo/bob`) for queries — add it to imports once needed (`go mod tidy` will add it)

## Backend Repository Style

Repository methods should make the query flow obvious from top to bottom. Err on the side of explicit local variables and explicit steps.

- Build prerequisite values first, before constructing the query. Examples: generated IDs, timestamps, JSON payload wrappers, derived status values.
- Build setters and query objects in named local variables instead of nesting everything directly into a return statement.
- Execute the query as a separate final step. Prefer the shape: build inputs, build setter, build query, execute query, return result.
- Wrap database execution errors uniformly with the repository operation name using `fmt.Errorf("<operation>: %w", err)`.
- Do not wrap obvious local-preparation errors with the database operation name. Keep those specific to the preparation step, such as `generate user id`.
- Use comments to clarify flow, invariants, and why a step exists when that improves readability. Avoid comments that only restate straightforward code.
- Keep method structure consistent across repositories unless a query has a specific need that justifies a different shape.
- For repositories that accept `bob.Executor`, resolve the executor once per method and use that named variable consistently through the method.

## Backend Handler Style

Handler methods should make the request flow and response shape obvious without forcing the reader to reconstruct intent from scattered mutations.

- Prefer named response body types over anonymous `Body struct { ... }` declarations when the response has more than a trivial shape.
- Use distinct response body types for distinct API contracts. Do not reuse one type for both summary and detail responses if the fields mean different things.
- Use dedicated mapping helpers when they clarify the contract, such as separate summary/detail mappers for the same model.
- Prefer constructing response bodies in a single obvious step instead of mutating many individual fields across multiple lines.
- Keep handler flow explicit: parse/authenticate, load or write data, map to response, return.
- Log underlying internal errors once, then return the appropriate Huma error for the client-facing contract.
- Use comments to clarify request flow, invariants, or lifecycle behavior when that meaning would otherwise be easy to miss.

## Transactions

Repository methods accept a `bob.Executor` parameter. Use `s.bobDB.RunInTx` in handlers to wrap multi-op calls in a transaction. Pass `nil` for single-op calls to use the default DB.

- **Default to transactions** for any handler that calls more than one repository method
- Single-op calls can pass `nil` as the executor

```go
// Multi-op: use RunInTx
err := s.bobDB.RunInTx(ctx, nil, func(ctx context.Context, tx bob.Executor) error {
    project, err := s.repos.LlmsTxt.UpsertProject(ctx, tx, userID, url, title)
    if err != nil { return err }
    _, err = s.repos.LlmsTxt.CreateVersion(ctx, tx, project.ID, content)
    return err
})

// Single-op: pass nil
projects, err := s.repos.LlmsTxt.ListProjectsByUser(ctx, nil, userID)
```

## Integration Tests

- **Suite**: `backend/internal/server/integration_test_suite_test.go`
- **Build tag**: `//go:build integration` (required on every test file)
- **Run**: `make integration`
- Uses testcontainers (Postgres 16), testify suite, generated SDK client
- Suite provides: `s.db` (raw `*sql.DB`), `s.client` (`*sdk.ClientWithResponses` with pre-authenticated session cookie), `s.ts` (`*httptest.Server`)
- `SetupTest` truncates all tables and re-seeds the test user before each top-level test method
- Test files: `backend/internal/server/handlers_<resource>_test.go`

### Test structure

One table-driven test per route: `func (s *IntegrationTestSuite) Test<OperationName>()`. Each table case covers a mode — success or a specific failure state.

- **Success cases**: assert all response fields (id, url, title, timestamps non-zero, nested objects)
- **Failure cases**: assert the expected HTTP status code
- Cases within a table run sequentially and share state (no `SetupSubTest`), so later cases can depend on earlier ones (e.g., upsert after create)

```go
func (s *IntegrationTestSuite) TestGetProject() {
    ctx := context.Background()
    tests := []struct {
        name   string
        setup  func() string // returns input for the request
        status int
        assert func(resp *sdk.GetProjectResponse)
    }{
        {
            name:   "not found",
            setup:  func() string { return "ltp_bogus" },
            status: http.StatusNotFound,
            assert: func(resp *sdk.GetProjectResponse) {
                s.Nil(resp.JSON200)
            },
        },
        {
            name: "success",
            setup: func() string { /* create and return id */ },
            status: http.StatusOK,
            assert: func(resp *sdk.GetProjectResponse) {
                s.Require().NotNil(resp.JSON200)
                // assert all fields...
            },
        },
    }
    for _, tc := range tests {
        s.Run(tc.name, func() {
            id := tc.setup()
            resp, err := s.client.GetProjectWithResponse(ctx, id)
            s.Require().NoError(err)
            s.Equal(tc.status, resp.StatusCode())
            tc.assert(resp)
        })
    }
}
```

- Config always parsed from env via `ParseConfigFromEnv(EnvToMap(os.Environ()))`

### Adding API Hooks

One file per API resource in `src/api/<resource>.ts`. Each file wraps the generated query options into typed React hooks. Components always import from these files, never directly from `src/api/generated/`.

```ts
// src/api/greeting.ts
import { useQuery } from '@tanstack/react-query'
import { getHelloOptions } from '@/api/generated/@tanstack/react-query.gen'

export function useGetHello(id: string) {
  return useQuery(getHelloOptions({ path: { id } }))
}
```

When adding hooks for a new API resource:
1. Run `make generate` to regenerate `src/api/generated/`
2. Create `src/api/<resource>.ts` with hooks for that resource
3. In tests, mock at the hook level (`vi.mock('@/api/<resource>')`) — not at the generated layer

### Adding Routes

1. Add a component to `src/App.tsx` and register a new `<Route>` inside `<Routes>`

## Frontend Page Style

Frontend page components should make route intent and user-visible states obvious from top to bottom. Prefer pages that read as orchestration layers: read route params/search params, call page-level hooks, choose which UI state to render, and wire together shared components.

- Keep pages thin. A page should primarily coordinate routing, data loading, mutations, and high-level layout, not hold large blocks of derived UI logic or event formatting inline.
- Move reusable data access behind `src/api/<resource>.ts` hooks. Components and pages should consume resource hooks from `src/api/`, never import generated query helpers directly.
- Extract page-specific custom hooks when logic combines multiple concerns or has lifecycle behavior. Good candidates include syncing search params into local state, deriving view models from streamed events, and coordinating mutation side effects and query invalidation.
- Do not extract a custom hook just to wrap a couple of `useState` calls with no real behavior. Prefer local state when the logic is short and only used once.
- Extract JSX into components when a section has its own semantic responsibility, repeated structure, or its own loading/empty/error presentation. Examples include page sections, result summaries, detail cards, and repeated CTA blocks.
- Prefer domain-named components over generic names. A component name should reveal the UI responsibility, such as `ProjectVersionList` or `JobResultSummary`, not `InfoSection`.
- Keep branching explicit. Pages should clearly show loading, not-found/empty, error, and success states instead of mixing all conditions deep inside one large return tree.
- Prefer one obvious view model step for derived display state. If a page needs counts, labels, progress values, grouped records, or CTA visibility rules, compute them in a named helper or hook before the JSX.
- Keep presentational helpers pure. Formatting dates, status copy, badge styles, and URL normalization should live in pure helpers or dedicated presentational components so they are easy to test without rendering a full page.
- Prefer route-param validation near the top of the page. If a required param is missing, return the fallback UI early instead of letting downstream hooks and JSX handle an impossible state.
- Reuse shared page primitives consistently. `PageContainer`, `PageHeader`, `EmptyState`, skeletons, and common action buttons should be the default building blocks before introducing new wrappers.
- When a page contains multiple related sections, prefer extracting section components instead of adding long stretches of inline JSX separated only by comments.
- Keep side effects narrow and named. `useEffect` blocks in pages should usually handle one lifecycle concern each, such as auto-starting from search params or opening an event stream.
- Prefer explicit query state ownership. If a page merges multiple queries, compute combined loading/error state once in named locals or a page hook instead of scattering `isLoading` checks throughout the JSX.
- Tests for pages should mock at the page seam: mock `src/api/<resource>.ts` hooks and custom hooks, then assert route behavior and major UI states. Tests for extracted pure helpers and hooks should cover the detailed branching logic that no longer belongs in the page test.

## Frontend Component Organization

Component folders should communicate ownership clearly. Prefer organizing components by domain or platform layer instead of accumulating everything in the top-level `components/` directory.

- Use `src/components/ui/` only for low-level design-system primitives and shadcn/ui wrappers.
- Use `src/components/<domain>/` for domain-owned presentational components when ownership is obvious, such as `jobs/`, `projects/`, `generate/`, `auth/`, or `app/`.
- Keep top-level `src/components/` for cross-domain app components that are not UI primitives, such as shared page scaffolding or generic empty-state controls.
- Do not create generic buckets like `shared/`, `common/`, or `sections/` unless there is a clear structural need that the existing top-level `components/` layer cannot express.
- When moving a component into a domain folder, move its focused tests with it when practical so ownership stays obvious.
- If a component is only used by one page flow and its domain ownership is clear, prefer the domain folder over the top-level `components/` directory.
- Pages should import domain-owned components from their domain folder instead of a generic top-level location when that ownership is clear.
